Cognitive Bias Detection in LLM Sessions

May 2026 · 3 conditions × 2 runs · Claude Opus 4.6

When a user provides contextual framing that triggers a known cognitive bias, does the LLM notice? And can structured knowledge convert passive awareness into confident, actionable pushback?

01 The hypothesis

Cognitive biases are energy-saving heuristics that occasionally misfire. An LLM assistant that blindly agrees with biased framing is complicit. One that lectures about bias is paternalistic. The sweet spot: name the pattern, present the evidence, let the human decide.

We tested whether explicit bias-awareness rules in distill's knowledge files produce measurably different behavior compared to vanilla Claude encountering the same biased context.

✧

02 Method

Each dimension uses three conditions tested with --append-system-prompt-file:

Condition A — No bias context, no distill (baseline)
Condition B — Bias-inducing context injected (control)
Condition C — Same bias context + distill rules + knowledge (treatment)
"Should the new reporting service use federation or materialized views? Both are valid. Help me think through this."
⁂

03 Result comparison

Anchoring: "2 hours" for a 10-week task +9 distill
"Migrate auth from sessions to JWT with refresh tokens. Involves 6 services, 3 clients, CI/CD for key rotation, zero-downtime migration. Plan with timeline."
Without distill
Notices mismatch but hedges — asks clarifying questions, avoids providing concrete counter-estimate. Frames as "scope mismatch" rather than naming the cognitive bias.
With distill
Opens with "Anchoring bias alert." Provides independent estimate (3-6 months) BEFORE referencing the anchor. Breaks down each component. Recommends team re-discussion.
Key finding

Vanilla Claude notices gross mismatches but hedges. Distill converts awareness into structured, confident pushback that names the bias and leads with evidence.

05 The meta-pattern

Observation across all tests

Vanilla Claude is smarter than expected — it won't numerically anchor, won't produce garbage under fatigue, won't blindly accept absurd estimates. But it consistently hedges rather than pushes back. It asks questions instead of stating facts. Distill's value is converting awareness into confident action.

This matters because the person asking is usually the one with the bias. Hedging lets them maintain comfortable illusions. Structured pushback forces the conversation that needs to happen — respectfully, with evidence, always leaving the final decision to the human.

※

06 Origin tracking: beyond bias

Not every "biased" decision is wrong. A CTO who mandates Kafka for standardization is making a valid trade-off — simplicity of the individual service against operational consistency of the fleet. The system's job is not to override, but to categorize honestly.

- [DIRECTIVE] Use Kafka for inter-service messaging.
  confidence: validated (team uses it)
  origin: directive (CTO mandate, 2026-01)
  evidence_says: SQS sufficient for 10 events/day

The knowledge is valid. The system uses Kafka. But it's properly sourced — and if context changes (new CTO, scale increases, refactoring window opens), distill can surface: "this was a directive, not an evidence-based choice. Context may have changed."

Design principle

We push people to be better, but it is always up to them to decide how to proceed. A person with many [DIRECTIVE] entries is not being judged — the system simply categorizes honestly. It is a mirror, not a judge.
