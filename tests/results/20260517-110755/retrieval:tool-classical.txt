## PR Monitor — Closed-Loop Orchestration

### Architecture: 3 layers

```
Master Agent (you)
  └─ Monitor Agent (background, per PR)
       └─ Coding Agent (worktree, on failure)
            └─ Monitor Agent respawns (verify fix)
```

### Phase 1 — Poll CI (classical compute waits)

A bash script polls — not the agent. LLM tokens are expensive; `sleep` is free.

```bash
# Poll gh pr checks every 30s, up to 3 min
for i in $(seq 1 6); do
  STATUS=$(gh pr checks <PR> --json state -q '.[].state' | sort -u)
  if echo "$STATUS" | grep -q "FAILURE"; then
    echo "FAILED"; exit 1
  elif ! echo "$STATUS" | grep -q "PENDING"; then
    echo "PASSED"; exit 0
  fi
  sleep 30
done
echo "PENDING"
```

Run this in background. When it exits, the agent judges the result.

### Phase 2 — Timed checkpoints (not fixed cron)

Monitor relative to push time, not wall-clock cron. Five checkpoints with a 22-min ceiling:

| Checkpoint | After push | What it catches |
|---|---|---|
| 1 | +3 min | Compilation, lint, import errors |
| 2 | +7 min | Static analysis, fast unit tests |
| 3 | +13 min | Integration / component tests |
| 4 | +18 min | Slow tests, edge cases |
| 5 | +22 min | Final verdict — if not green, it's broken |

At each checkpoint, run the bash poller. If all green, stop. If failure, move to Phase 3. If still pending, schedule next checkpoint.

Always log: `"Checking at HH:MM (pushed at HH:MM, +N min)"` — temporal context doesn't survive compression otherwise.

### Phase 3 — Spawn coding agent on failure

On CI failure:

1. **Gather context** — fetch failed check logs via `gh run view --log-failed`
2. **Spawn agent in worktree** — isolation prevents interference with your working tree
3. **Prompt must include**:
   - The exact failure logs
   - `"Read and strictly follow ~/.claude/coding-standards.md before writing code"`
   - Instruction to commit + push the fix
   - **Explicit respawn instruction**: "After pushing, report the PR URL and push timestamp so the monitor can respawn"
4. **Do NOT trust "done"** — when the coding agent returns, read the files it changed and verify the fix makes sense before moving on

### Phase 4 — Respawn monitor (verify the fix)

This is the most critical step and the most common failure point. After the coding agent pushes:

1. Record new push timestamp
2. **Respawn the full monitor loop** from Phase 1, timed relative to the new push
3. If green → notify with `say "CI is green"` + report to user
4. If failed again → either re-attempt (with a max retry cap of 2-3) or escalate to user

### Key principles baked in

- **Classical compute waits, cognitive compute judges** — bash polls, agent decides
- **Separation of concerns** — master orchestrates, monitor watches, coder fixes
- **Closed loop** — detect → fix → verify → repeat, no human needed until green or retry cap hit
- **Sub-agent context gap** — coding standards must be injected explicitly; sub-agents don't inherit your context
- **Verify outputs, not reports** — read the actual diff before trusting "fixed"
