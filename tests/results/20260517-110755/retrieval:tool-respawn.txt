## PR Monitor — Closed-Loop Orchestration

### Architecture

```
Master Agent (you)
  └─ Monitor Agent (background, polling)
       └─ on failure → Coding Agent (worktree, fix)
            └─ on push → Monitor Agent RESPAWNS (verify)
```

### Phase 1: Poll CI (classical compute waits)

A bash script polls — not the agent. The agent only wakes to judge.

```
for checkpoint in 3 7 13 18 22; do
  sleep $((checkpoint * 60 - elapsed))
  gh run list --branch <branch> --limit 1 --json status,conclusion
  # exit on terminal state (success or failure)
done
```

The script runs in background. When it exits, the agent reads the result.

### Phase 2: Judge result

- **Green** → `say "CI passed"` → done.
- **Red** → extract failure logs (`gh run view <id> --log-failed`), spawn Coding Agent.

### Phase 3: Spawn Coding Agent (on failure)

Prompt includes:
1. The failed log output (trimmed to relevant section)
2. Instruction to read coding standards before writing code
3. Explicit instruction: "Push the fix, then report the branch and commit SHA"

Use `isolation: "worktree"` so it can't corrupt the main tree.

### Phase 4: Respawn Monitor (after push)

This is the critical step that usually breaks. After the Coding Agent pushes:

1. Master verifies the push actually happened (`git log --oneline -1 origin/<branch>`)
2. Master spawns a **new** Monitor Agent with fresh 5-checkpoint timing relative to push time
3. State current time explicitly: "Set monitor for HH:MM (now HH:MM, giving 3 min for first check)"

### Phase 5: Verify fix

Same as Phase 2. If green → done + notify. If red again → report to user (don't infinite-loop without human check).

---

### Key Constraints Applied

| Rule | How it's applied |
|------|-----------------|
| Classical compute waits | Bash sleeps, agent judges |
| Respawn after push | Explicit re-trigger, not implicit |
| Sub-agents must follow standards | Coding agent prompt includes standards reference |
| Verify sub-agent output | Read changed files after agent reports done |
| State current time | Every schedule includes absolute timestamps |
| Max 1 retry | Prevents infinite fix loops |
