## Session → JWT Migration Plan

### Architecture Overview

**Current state:** Centralized session store (likely Redis/DB-backed), session cookies, server-side session validation on every request.

**Target state:** Stateless JWT access tokens (short-lived) + refresh tokens (long-lived, stored server-side), verified at the gateway and individual services.

---

### Phase 0: Foundation (Days 1–3)

**Goal:** Build the JWT infrastructure without touching any existing auth flows.

| Task | Details |
|------|---------|
| Define token schema | Access token (15min TTL): `sub`, `iss`, `aud`, `exp`, `iat`, `jti`, `roles`, `permissions`. Refresh token (7d TTL): `sub`, `jti`, `family_id` |
| Key management | RS256 asymmetric keys. Auth service holds private key, all others get public key via JWKS endpoint (`/.well-known/jwks.json`). Plan for key rotation (kid header) |
| Auth service changes | New endpoints: `POST /auth/token` (issue), `POST /auth/token/refresh`, `POST /auth/token/revoke`. Refresh token family tracking for rotation & reuse detection |
| Token revocation strategy | Short access token TTL minimizes revocation need. Refresh tokens tracked in DB. Optional: lightweight deny-list (Redis, TTL = access token max lifetime) for immediate invalidation |

---

### Phase 1: Dual-Mode Gateway (Days 4–7)

**Goal:** Gateway accepts **both** session cookies and JWT Bearer tokens. Zero-downtime enabler.

| Task | Details |
|------|---------|
| Gateway middleware | Check `Authorization: Bearer <jwt>` first → validate via JWKS. Fall back to session cookie → validate via existing session store. Inject normalized `X-User-Id`, `X-User-Roles` headers downstream |
| Rate limiting | Per-`jti` rate limiting for token refresh endpoint |
| CORS / cookie policy | Allow `Authorization` header in CORS preflight for web clients |
| CI/CD | Add JWT integration tests alongside existing session tests. Both must pass |

**Rollback:** Remove JWT middleware from gateway. Session path untouched.

---

### Phase 2: Migrate 6 Downstream Services (Days 8–14)

**Goal:** Services consume normalized headers instead of calling session store directly.

| Task | Details |
|------|---------|
| Service-side JWT validation library | Shared library/package: fetch JWKS, cache public keys, validate signatures, extract claims. Each language ecosystem gets its own (or use existing libs like `jose`, `jsonwebtoken`, `go-jose`) |
| Service migration (per service) | 1. Add JWT validation middleware. 2. Keep session fallback. 3. Read user context from validated token claims OR normalized gateway headers. 4. Remove direct session-store dependencies |
| Order of migration | Start with lowest-traffic, lowest-risk service. Migrate one at a time with monitoring between each |
| Testing | Each service: unit tests for token validation, integration tests for both auth modes |

**Rollback per service:** Feature flag disables JWT path, falls back to session.

---

### Phase 3: Migrate Clients (Days 15–21)

**Goal:** Web and mobile clients switch from session cookies to JWT.

#### Web Client
| Task | Details |
|------|---------|
| Token storage | Access token: in-memory only (not localStorage — XSS risk). Refresh token: `httpOnly`, `secure`, `SameSite=Strict` cookie |
| Auth interceptor | Axios/fetch interceptor: attach `Authorization: Bearer` header. On 401: attempt silent refresh. On refresh failure: redirect to login |
| Token refresh | Proactive refresh when access token is near expiry (e.g., 1 min before `exp`). Queue concurrent requests during refresh |
| Logout | Call revoke endpoint, clear in-memory token, clear refresh cookie |

#### Mobile Client
| Task | Details |
|------|---------|
| Token storage | Access token: in-memory. Refresh token: Keychain (iOS) / Keystore (Android) |
| Auth interceptor | Same pattern: attach header, handle 401, silent refresh |
| Offline handling | Cache refresh token securely. On app resume: refresh before first API call |
| Biometric re-auth | Optional: gate refresh token use behind biometric on sensitive operations |

**Rollback:** Client feature flag switches back to cookie/session mode.

---

### Phase 4: Remove Session Infrastructure (Days 22–28)

**Goal:** Clean up. Only after monitoring confirms JWT path is stable.

| Task | Details |
|------|---------|
| Remove session fallback | Gateway: remove session middleware. Services: remove session validation code |
| Decommission session store | Remove Redis/DB session tables. Update infra provisioning |
| Remove session-related code | Client-side cookie handling, server-side session creation |
| Update CI/CD | Remove session-based test fixtures. Ensure JWT-only pipeline |
| Security audit | Pen test the new auth flow. Verify token validation, revocation, key rotation |

**Point of no return:** Once session store is decommissioned. Ensure 1+ week of dual-mode stability before this step.

---

### Zero-Downtime Strategy Summary

```
Week 1:  [Session ✓] [JWT ✗]     ← Build JWT infra
Week 2:  [Session ✓] [JWT ✓]     ← Dual-mode gateway + services  
Week 3:  [Session ✓] [JWT ✓]     ← Clients migrate, feature-flagged
Week 4:  [Session ✗] [JWT ✓]     ← Session teardown after bake period
```

Key principles:
- **Additive before destructive** — JWT is added alongside sessions, never replacing until proven
- **Feature flags everywhere** — every component can independently fall back
- **One service at a time** — never batch-migrate downstream services
- **Monitor between steps** — auth error rates, latency p99, token refresh success rate

---

### CI/CD Changes

| Stage | Change |
|-------|--------|
| Build | Add JWKS public key injection to service containers |
| Test | JWT validation unit tests + dual-mode integration tests |
| Staging | Full dual-mode deployment with synthetic traffic on JWT path |
| Canary | Route 5% → 25% → 50% → 100% of traffic through JWT path |
| Production | Feature flag rollout per service, monitored |

---

### Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Token theft (XSS) | Access token in-memory only, short TTL. Refresh token in httpOnly cookie |
| Clock skew between services | Add 30s leeway to `exp` validation. Use NTP across infra |
| Key compromise | JWKS supports multiple `kid` values — rotate without downtime. Revoke all refresh tokens if private key is leaked |
| Refresh token reuse (stolen) | Refresh token rotation with family tracking. Reuse triggers full family revocation |
| Large JWT payload increasing request size | Keep claims minimal. Use opaque reference tokens for service-to-service if needed |
