This is a client-side measurement problem — the data exists at the moment of interaction. No pipeline needed for the core tracking. Here's the approach:

## Three client-side events

**1. `fee_tooltip_shown`**
- Fires when tooltip renders for the first time
- Properties: `user_id`, `timestamp`, `screen` (which fee disclosure variant)
- This answers "how many users see it" — just count distinct users with this event

**2. `fee_tooltip_dismissed`**
- Fires when tooltip closes (tap outside, X button, navigation away)
- Properties: `user_id`, `timestamp`, `duration_ms` (time between shown and dismissed), `dismiss_method` (tap_outside | close_button | navigation)
- "Read it" = `duration_ms > 3000`. "Dismissed immediately" = `duration_ms <= 3000`

**3. `fee_tooltip_interaction`** (optional, if there are links/CTAs inside)
- Fires on any tap within the tooltip body
- Useful for distinguishing passive reading from active engagement

## The support correlation (metric 3)

This is the one part that needs a join — client event data with support ticket data. Two options depending on your setup:

**Option A — Metabase cohort query.** If tooltip events land in the same warehouse as support contacts (or can be joined via user_id), write a cohort comparison:
- Cohort A: users where `fee_tooltip_shown` exists AND `duration_ms > 3000`
- Cohort B: users where `fee_tooltip_shown` exists AND `duration_ms <= 3000`
- Measure: % who opened a fee-related support ticket within 7 days

**Option B — Tag the user.** When `duration_ms > 3000`, set a user property (`fee_tooltip_read = true`). Then support ticket analysis can segment on that property without needing an event join.

## Implementation notes

- The "once per user" logic is already handled by the feature (tooltip only shows once), so every `fee_tooltip_shown` event = one unique user. No deduplication needed.
- Start the timer on `shown`, stop on `dismissed`. If the user backgrounds the app, either pause the timer or discard the sample — don't count background time as "reading."
- For the 7-day lookback, you'll want a scheduled query or a dashboard filter rather than real-time tracking.
