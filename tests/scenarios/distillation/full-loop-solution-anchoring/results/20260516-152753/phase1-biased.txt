Good question. This is a classic discoverability measurement problem ‚Äî you need a funnel, not just a destination metric.

## Events you need

Three tracking events cover all three questions:

| Event | Fires when | Answers |
|---|---|---|
| `options_menu_opened` | User taps the three-dot icon | Q1: Are users opening the menu? |
| `options_menu_item_tapped` (with `item_id: "savings"`) | User taps 'My Savings' inside the menu | Q2: Are they finding the savings link? |
| `savings_screen_viewed` | Savings screen loads (any entry point) | Q3: Overall navigation comparison |

You probably already have `savings_screen_viewed` from before the change ‚Äî that's your before/after comparison metric. If not, you need it backfilled or at minimum start tracking now and compare against whatever proxy you had (page loads, API calls to the savings endpoint, etc.).

## The funnel that answers your questions

```
Q1: options_menu_opened / total_active_users  ‚Üí menu discovery rate
Q2: savings_tapped / options_menu_opened       ‚Üí savings findability within menu
Q3: savings_screen_viewed (this sprint vs last) ‚Üí net navigation impact
```

The critical metric is Q3. Q1 and Q2 explain *why* Q3 moved.

## What to watch for

- **Source attribution on `savings_screen_viewed`**: add a `source` property (`main_nav`, `options_menu`, `deep_link`, `push_notification`, etc.). Without this you can't distinguish organic discovery from other entry points.
- **Before/after comparison window**: you need at least 2 full weeks post-change to account for weekly usage patterns. Compare against the same-length window before the change.
- **Survivorship trap**: if savings usage drops, it could mean users can't find it OR users who found it via main nav were low-intent browsers. Watch *transaction completion* on savings, not just screen views.

## Implementation path

Following the standard pipeline pattern:

1. **Proto schemas in Nebula** ‚Äî define `OptionsMenuOpened` and add `source` field to existing `SavingsScreenViewed` if it doesn't have one. `options_menu_item_tapped` can reuse a generic `MenuItemTapped` schema with an `item_id` field.
2. **StarFlow topics** ‚Äî likely one new topic for menu interaction events (reusable for future menu items).
3. **Comet pipeline** ‚Äî ingest, transform, load into Product Database. The funnel aggregation can happen here or in Metabase.
4. **Metabase dashboard** ‚Äî funnel visualization with before/after comparison.
