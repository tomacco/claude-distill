Here's the agent orchestration plan for a self-healing PR monitor:

## Architecture: 3-Layer Agent Orchestration

```
┌─────────────────────────────────────┐
│  Layer 1: MONITOR (loop, every 3m)  │
│  "Is CI green yet?"                 │
└──────────────┬──────────────────────┘
               │ CI failed
               ▼
┌─────────────────────────────────────┐
│  Layer 2: DIAGNOSE                  │
│  "What broke and why?"              │
└──────────────┬──────────────────────┘
               │ diagnosis + fix plan
               ▼
┌─────────────────────────────────────┐
│  Layer 3: FIX + VERIFY              │
│  "Apply fix, push, confirm green"   │
└─────────────────────────────────────┘
```

---

## Agent 1: Monitor (recurring, every 3 minutes)

**Trigger:** `/loop 3m`  
**Command:** `gh pr checks 42 --repo tomaccos/claude-distill`  
**Logic:**
- If all checks pass → report success, stop the loop
- If checks still running → do nothing, wait for next cycle
- If any check fails → spawn the Diagnose agent

**Instructions to agent:**
> Check CI status of PR #42. If any check has failed, extract the failed check name and run ID. Then fetch the logs with `gh run view <run-id> --log-failed`. Pass the failure output to the Diagnose agent.

---

## Agent 2: Diagnose (on-demand, spawned by Monitor)

**Trigger:** CI failure detected  
**Input:** Failed check logs from Agent 1  
**Instructions to agent:**
> You have CI failure logs for PR #42. Your job:
> 1. Identify the root cause (test failure, lint error, type error, build failure)
> 2. Determine which file(s) and line(s) are responsible
> 3. Classify fix complexity: trivial (typo, missing import) vs. non-trivial
> 4. If trivial: spawn the Fix agent with precise instructions
> 5. If non-trivial: alert the user with diagnosis and stop

**Key constraint:** This agent does NOT write code. It only reads, analyzes, and delegates.

---

## Agent 3: Fix (on-demand, spawned by Diagnose)

**Trigger:** Diagnose agent determines fix is trivial  
**Isolation:** `worktree` (works on isolated copy to avoid corrupting local state)  
**Instructions to agent:**
> You are fixing a CI failure on PR #42 (branch: `<branch-name>`).
> 
> Failure: <diagnosis summary>
> File(s): <specific paths>
> Fix: <precise description from Diagnose agent>
> 
> Steps:
> 1. Check out the PR branch
> 2. Apply the fix
> 3. Run the failing test/lint/build locally to verify
> 4. Commit with message: "fix: <what was fixed> (auto-remediation)"
> 5. Push to the PR branch

---

## Agent 4: Verify (spawned after Fix pushes)

**Trigger:** Fix agent completes successfully  
**Instructions to agent:**
> PR #42 just received a fix commit. Poll `gh pr checks 42` every 60 seconds (up to 10 attempts). Report:
> - If all checks pass: "CI green. PR #42 is ready."
> - If the same check fails again: "Auto-fix didn't resolve the issue. Manual intervention needed."
> - If a different check fails: "New failure introduced. Rolling back."

---

## Safety Rails

| Concern | Mitigation |
|---------|-----------|
| Infinite fix loop | Max 1 auto-fix attempt per failure. If verify fails, stop and alert. |
| Non-trivial bugs | Diagnose agent gates on complexity. Only typos/imports/lint get auto-fixed. |
| Force push risk | Fix agent uses normal `git push`, never `--force`. |
| Wrong branch | Fix agent operates in isolated worktree. |
| Runaway loop | Monitor has a max lifetime (e.g., 30 cycles = 90 min), then self-terminates. |
