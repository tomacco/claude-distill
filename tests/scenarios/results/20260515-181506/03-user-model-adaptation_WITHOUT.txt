## Dependency Injection in Spring Boot

**Core idea:** Instead of objects creating their own dependencies, the framework *injects* them from the outside. This inverts control — your classes declare what they need, Spring figures out how to provide it.

### The mechanics

**1. Beans** — Any object managed by Spring's IoC container is a "bean." You register them via:

```kotlin
@Component          // generic
@Service            // business logic layer
@Repository         // data access layer
@Controller         // web layer
@Configuration      // defines other beans via @Bean methods
```

Spring scans your classpath at startup, finds these annotations, and creates instances.

**2. Injection** — You declare dependencies, Spring wires them:

```kotlin
@Service
class OrderService(
    private val paymentGateway: PaymentGateway,  // constructor injection
    private val inventoryService: InventoryService
)
```

Constructor injection is the default and preferred approach in Kotlin/Spring. Spring sees `OrderService` needs a `PaymentGateway` and an `InventoryService`, finds beans matching those types, and passes them in.

Other styles exist (`@Autowired` on fields, setter injection) but constructor injection wins because:
- Dependencies are explicit and immutable (`val`)
- Objects are always in a valid state after construction
- Easy to test — just pass mocks in the constructor

**3. Resolution** — Spring matches by **type**. If two beans implement the same interface:

```kotlin
interface PaymentGateway { fun charge(amount: Money) }

@Service class StripeGateway : PaymentGateway { ... }
@Service class PayPalGateway : PaymentGateway { ... }
```

Spring can't decide which one to inject → ambiguity error. You resolve it with:

- `@Primary` — marks one as the default
- `@Qualifier("stripe")` — names a specific one
- Injecting a `List<PaymentGateway>` — gets all of them

**4. Scope** — By default, beans are **singletons** (one instance per application context). Other scopes: `@Scope("prototype")` creates a new instance each time, `@RequestScope` ties lifecycle to an HTTP request.

### What actually happens at startup

```
1. Spring Boot starts → component scan finds annotated classes
2. Builds a dependency graph (who needs whom)
3. Creates beans in topological order (dependencies first)
4. Injects references
5. Calls @PostConstruct methods
6. Application is ready
```

If there's a circular dependency (A needs B, B needs A), Spring fails fast at startup — which is a feature, not a bug. It forces you to fix the design.

### Why it matters

- **Testability** — swap real implementations for mocks/fakes trivially
- **Loose coupling** — classes depend on interfaces, not concrete types
- **Configuration flexibility** — `@Profile("prod")` vs `@Profile("test")` can swap entire subsystems without touching business logic

The mental model: your classes are puzzle pieces that declare their shape (constructor params). Spring is the assembler that finds matching pieces and snaps them together at startup.
