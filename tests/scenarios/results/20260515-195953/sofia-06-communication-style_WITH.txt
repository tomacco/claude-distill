Use `http.Server.Shutdown(ctx)` with OS signal trapping:

```go
func main() {
    srv := &http.Server{Addr: ":8080", Handler: mux}

    // Start server in a goroutine
    go func() {
        if err := srv.ListenAndServe(); err != http.ErrServerClosed {
            log.Fatalf("listen: %v", err)
        }
    }()

    // Wait for interrupt signal
    quit := make(chan os.Signal, 1)
    signal.Notify(quit, syscall.SIGINT, syscall.SIGTERM)
    <-quit

    // Give active connections a deadline to finish
    ctx, cancel := context.WithTimeout(context.Background(), 30*time.Second)
    defer cancel()

    if err := srv.Shutdown(ctx); err != nil {
        log.Fatalf("forced shutdown: %v", err)
    }
    log.Println("server exited cleanly")
}
```

**Key points:**

- `Shutdown` stops accepting new connections and waits for in-flight requests to complete (up to the context deadline).
- `ListenAndServe` returns `http.ErrServerClosed` after `Shutdown` — that's expected, not an error.
- The timeout (30s here) should match your infra's drain period (e.g., Kubernetes `terminationGracePeriodSeconds`).
- If you have background workers, DB connections, etc., close them **after** `Shutdown` returns — the server is done at that point.

**Common pitfalls to avoid:**

- **Buffered signal channel** — use `make(chan os.Signal, 1)`, not unbuffered. An unbuffered channel can miss the signal if nobody is receiving yet.
- **Health checks** — if behind a load balancer, start returning unhealthy *before* or *at* signal receipt so the LB stops routing traffic during the drain window.
- **Long-lived connections** (WebSockets, SSE) — `Shutdown` won't forcibly close them. Use `srv.RegisterOnShutdown(fn)` to notify those handlers, or track them and close manually.
