Use `signal.NotifyContext` + `srv.Shutdown()`:

```go
func main() {
    ctx, stop := signal.NotifyContext(context.Background(), syscall.SIGINT, syscall.SIGTERM)
    defer stop()

    srv := &http.Server{
        Addr:    ":8080",
        Handler: newRouter(),
    }

    // Start server in goroutine
    go func() {
        if err := srv.ListenAndServe(); err != nil && err != http.ErrServerClosed {
            log.Fatalf("listen: %v", err)
        }
    }()

    // Block until signal
    <-ctx.Done()
    log.Println("shutting down...")

    // Give in-flight requests a deadline to finish
    shutdownCtx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    if err := srv.Shutdown(shutdownCtx); err != nil {
        log.Fatalf("forced shutdown: %v", err)
    }

    log.Println("server stopped cleanly")
}
```

Key points:

- **`srv.Shutdown(ctx)`** stops accepting new connections and waits for in-flight requests to complete (or until the context deadline)
- **`signal.NotifyContext`** (Go 1.16+) is cleaner than raw `signal.Notify` + channel
- **10s timeout** is a good default — tune it based on your slowest expected request (your payment webhook timeout is 30s, so you might want longer for that service)
- **Don't use `srv.Close()`** — it kills connections immediately with no drain
- If you have background workers (Kafka consumers, Redis connections), close them **after** `srv.Shutdown` returns, in reverse initialization order

For Chi specifically, nothing special is needed — Chi handlers respect the request context, so cancellation propagates naturally.
