A few things I'd flag:

**Security**
- **No signature verification.** Anyone can POST to this endpoint and fake events. You need to validate an HMAC/signature header from the webhook provider before processing.
- **No body size limit.** `io.ReadAll(r.Body)` will read until EOF — a malicious client can send gigabytes. Use `io.LimitReader(r.Body, maxBytes)`.

**Correctness**
- **No idempotency check.** If the provider retries a webhook (and they will), you'll overwrite `CompletedAt`/`FailedAt` with a new timestamp and re-send notifications. Check for a delivery ID or event ID you've already processed.
- **Race condition on payment state.** Two concurrent webhooks for the same payment (e.g. `completed` then `failed` retry) do a read-modify-write without any optimistic locking or transaction. One update can silently clobber the other.

**Minor**
- `time.Now()` for `CompletedAt`/`FailedAt` records when *you* processed it, not when the event actually occurred. If the event payload has a timestamp, prefer that.
- `payment.failed` case doesn't notify the user — intentional? Failed payments usually warrant a notification too.
- Returning `200` for unknown event types is fine (providers expect it), but you might want to return `200` earlier for known-but-unprocessable events too, since returning `500` will cause retries — which hits the idempotency gap above.

The signature verification and body size limit are the most urgent. The idempotency gap will bite you in production once the provider starts retrying.
