A few things I'd flag:

**Security**
- **No signature verification.** This is the big one. Anyone can POST to this endpoint and fake events. You need to validate an HMAC signature (or whatever your provider uses) from a request header before processing anything.
- **No body size limit.** `io.ReadAll(r.Body)` on an unbounded body is a DoS vector. Use `http.MaxBytesReader` to cap it.

**Correctness**
- **No idempotency.** If the provider retries a webhook (and they will), you'll overwrite `CompletedAt`/`FailedAt` with a new timestamp and re-send notifications. Check if the payment is already in the target status before mutating.
- **Race condition on payment.** `GetPayment` then `UpdatePayment` is a read-modify-write without any locking or optimistic concurrency. Two concurrent webhooks for the same payment can clobber each other.

**Minor**
- `payment.failed` doesn't notify the user — intentional? Usually you'd want to notify on failures too.
- Unknown event types return 200. That's fine (most providers recommend it), just make sure it's intentional so you don't silently swallow events you meant to handle.

The signature verification is the critical one — everything else is secondary until that's in place.
