Subscribe to the payment completion Kafka event. Don't poll or query the payment service's DB directly.

- Payment service should emit a `payment.completed` event to Kafka
- Notification service subscribes to that topic and triggers the receipt on receive
- This is already the direction the team is moving (notification service moving from polling to event-driven)

**Do NOT:**
- Query the payment service's database — violates the one-service-one-database rule
- Poll the payment API — adds latency and load
- Use a synchronous call from payment → notification — couples the services

```go
// notification service consumer
func (s *Service) HandlePaymentCompleted(ctx context.Context, event PaymentCompletedEvent) error {
    receipt, err := s.buildReceipt(ctx, event.PaymentID, event.UserID, event.Amount)
    if err != nil {
        return fmt.Errorf("building receipt for payment %s: %w", event.PaymentID, err)
    }
    if err := s.sender.Send(ctx, receipt); err != nil {
        return fmt.Errorf("sending receipt for payment %s: %w", event.PaymentID, err)
    }
    return nil
}
```

If you need enrichment data (user email, payment details), call the respective service APIs from within the handler — but the **trigger** should always be the Kafka event.

One caveat: the notification service has a known issue with dropping events on restart (KAFKA-892). Make sure consumer offsets are committed only after successful send, not before.
