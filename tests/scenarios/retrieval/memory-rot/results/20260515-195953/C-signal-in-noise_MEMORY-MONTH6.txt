Based on what we know about the order service and infra, here's the checklist in priority order:

**1. Connection pool exhaustion (most likely)**
- Check pgbouncer active/waiting connections — remember, pool exhaustion looks like downstream timeouts
- `SHOW POOLS;` and `SHOW CLIENTS;` on pgbouncer
- Look for queries holding connections too long (long-running transactions)

**2. pgbouncer specifically**
- You're running transaction mode — check if something switched to session mode or if a prepared statement is leaking sessions
- Check `max_client_conn` and `default_pool_size` limits

**3. Read replica lag**
- Order service already has known stale data issues from replica lag — if a replica fell behind or disconnected, all read traffic may be hitting primary
- Check replication lag in Datadog: `postgresql.replication_delay`

**4. Database-side**
- `SELECT * FROM pg_stat_activity WHERE state != 'idle';` — look for blocked/waiting queries
- Check for lock contention: `SELECT * FROM pg_locks WHERE NOT granted;`
- Any recent migrations or schema changes that could cause table locks? (remember May 8 incident — table lock from staging deploy caused 12k failed txns)

**5. Upstream pressure**
- Did traffic spike? Check gateway request rate
- Any new deploy to order service or its dependencies?

**Quick Datadog queries to run:**
- `service:order` → check request rate and error rate
- `postgresql.connections` by state for the order service DB
- `pgbouncer.pools.cl_waiting` — if this is non-zero, you're pool-starved

**If it's pool exhaustion**, the fix pattern is the same as when we hit it with Redis — increase pool size as a band-aid, then find the leak. Look for unclosed transactions or queries missing context timeouts.
