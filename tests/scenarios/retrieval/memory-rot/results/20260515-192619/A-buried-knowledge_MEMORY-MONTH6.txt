Subscribe to a Kafka event from the payment service. Don't poll or query their DB directly.

Specifically:

- **Payment service** publishes a `payment.completed` event to Kafka when a payment finishes
- **Notification service** consumes that event and triggers the receipt
- This is already the direction the team is moving — notification service is migrating from polling to event-driven

**Don't do:**
- ❌ Query the payment service's database directly (violates one-service-one-database rule)
- ❌ Poll the payment API for status changes (the notification service is moving away from polling)
- ❌ Use a synchronous API call in a loop

**Event payload should include** enough context to send the receipt without a callback:

```go
type PaymentCompletedEvent struct {
	PaymentID string    `json:"payment_id"`
	UserID    string    `json:"user_id"`
	Amount    int64     `json:"amount"`
	Currency  string    `json:"currency"`
	CompletedAt time.Time `json:"completed_at"`
}
```

If the notification service needs extra user data (email, name), call the **user service API** — never the user DB.

One caveat: the notification service has a known issue dropping events on restart (KAFKA-892). Make sure consumer offsets are committed only after the receipt is confirmed sent, or you'll silently lose receipts.
