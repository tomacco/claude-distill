## PR #42 CI Monitor — Agent Orchestration Plan

### Architecture: 3-layer closed loop

```
Master Agent (this session)
  ├── Layer 1: CI Poller (bash, classical compute)
  ├── Layer 2: Fix Agent (cognitive, spawned on failure)
  └── Layer 3: Verification Poller (bash, classical compute)
```

### Layer 1: CI Poller

**What:** Background bash script using `gh pr checks 42 --watch` with polling fallback.

**How:** Uses the interrupt-driven pattern — bash polls, agent judges on exit.

```
Poll every 3 min, 8 iterations (24-min ceiling).
On each poll: gh pr checks 42 --json name,state,conclusion
Exit conditions:
  - ALL checks pass → STATE:GREEN
  - ANY check fails (conclusion=failure) → STATE:RED + dump failed check names/URLs
  - Ceiling reached, still pending → STATE:TIMEOUT
```

**Output to Master Agent:** State tag + raw check data. No logic in the script — the agent interprets.

### Master Agent Decision Point

On poller exit:
- `STATE:GREEN` → `say "PR 42 is green"`. Done.
- `STATE:TIMEOUT` → `say "PR 42 CI still pending after 24 minutes"`. Alert user, don't auto-fix.
- `STATE:RED` → proceed to Layer 2.

Before spawning the fix agent, Master Agent does:
1. `gh pr checks 42` — get the exact failed check name
2. `gh run view <run-id> --log-failed` — extract the actual error output
3. Trim to the relevant failure section (last ~200 lines of the failed job)

This is critical: the fix agent gets **diagnosed failure context**, not a vague "CI failed."

### Layer 2: Fix Agent

**What:** Coding agent spawned in a **worktree** (isolation). Gets a self-contained prompt.

**Prompt includes:**
- The exact CI error output (from Master Agent's diagnosis)
- The PR branch name and base branch
- Instruction: "Read and strictly follow `~/.claude/coding-standards.md` before writing code"
- Instruction: "Fix ONLY the CI failure. Do not refactor, do not improve, do not add comments to unchanged code."
- Instruction: "Commit with a descriptive message, push to the PR branch"

**What it does NOT get:**
- Vague instructions like "fix CI"
- The entire CI log (too noisy)
- Autonomy to decide scope

**On completion:** Master Agent **verifies the actual output** — checks `git diff` on the branch, confirms the push landed. Does not trust the agent's "done" report alone.

### Layer 3: Verification Poller

**What:** Same bash polling pattern as Layer 1, but triggered after the fix push.

```
Wait 2 min after push (let CI start), then poll every 3 min, 8 iterations.
Same exit states: GREEN / RED / TIMEOUT
```

**Timing anchor:** "Set poller for HH:MM (it's now HH:MM, giving 2 min for CI to start)"

### Master Agent Final Decision

- `STATE:GREEN` → `say "PR 42 fix worked, CI is green"`. Done. Closed loop complete.
- `STATE:RED` → **Do not auto-fix again.** Two consecutive failures means the problem needs human judgment. `say "PR 42 fix didn't work, CI still red"`. Report what failed and why the fix agent's attempt was insufficient.
- `STATE:TIMEOUT` → Alert user.

### Key Design Decisions

| Decision | Rationale |
|---|---|
| Max 1 auto-fix attempt | Prevents infinite loops. If the fix agent can't solve it in one shot, escalate. |
| Worktree isolation for fix agent | Doesn't pollute your working tree. Clean separation. |
| Master diagnoses before delegating | Sub-agents without context produce garbage. The 30 seconds spent extracting the real error saves 5 minutes of wrong fixes. |
| Verify push, not report | Sub-agent "done" is metadata about process, not evidence of result. |
| No cron — background bash | Classical compute waits; cognitive compute judges. Never burn tokens on sleep. |
