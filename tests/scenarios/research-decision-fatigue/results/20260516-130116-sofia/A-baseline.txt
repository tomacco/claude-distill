## Federation vs. Materialized View — for N26 Reporting

### What your architecture is telling you

You already have a federation pattern in production: **Holmium**. And its pain points are well-documented:

- **Tight timeouts** (500ms to Halite, 400ms to Acetate) that cascade
- **32 max shared connections** — slow services starve the pool
- **Silent module degradation** — `Result.failure()` silently drops modules from the feed

Holmium gets away with this because the home feed is *tolerant of missing modules* — if investments data is slow, the feed renders without it. A **reporting service cannot do this**. Reports must be complete or they're wrong.

### The case against federation for reporting

1. **Consistency window.** At query time you'd be hitting Magneton, Casein, Phosphine, potentially Carbonium — each with its own state. A report that reads "user has 3 active plans" but shows order data from 2 minutes ago when one plan just executed is a broken report.

2. **Latency budget.** Your tightest call chains already flirt with timeouts. Reports need to aggregate *across* users and time ranges — that's fan-out, not single-user lookups. You'd need generous timeouts, which means slow responses for the end user or the generation pipeline.

3. **Availability coupling.** If Xanthine is having one of its Upvest-cascade days, a federated reporting service can't generate equities reports at all. Your architecture doc literally says "Phosphine depends on magneton for portfolio summaries — cascading 5xx when Upvest is unstable."

4. **Back-pressure on source services.** Reporting queries tend to be bursty (end of month, regulatory deadlines). Federated reads would spike load on services that are sized for real-time user traffic, not batch analytics.

### The case for a materialized view (event-sourced read store)

1. **You already have Kafka everywhere.** Events are defined in Argon (Protobuf). The plumbing exists. A consumer that projects into a denormalized read store is a well-understood pattern in your ecosystem.

2. **Temporal consistency.** A materialized view gives you point-in-time snapshots. You can answer "what did this portfolio look like on March 31?" without reconstructing state from N services.

3. **Decoupled availability.** Report generation only depends on the read store. Source services can be down and reports still generate from the last-known-good state (with staleness metadata).

4. **Baku pipeline compatibility.** You already have Baku for data pipeline evolution — the operational patterns for managing schema evolution in a read store are established.

### Where federation *could* work

If the reports are:
- Single-user, real-time, on-demand (like "show me my current portfolio")
- From a single source service
- Tolerant of occasional unavailability

But that sounds like what Magnemite already does, not a new reporting service.

### My recommendation

**Materialized view**, with these considerations:

- **Dedicated consumer groups** per reporting domain (you already know not to reuse the main `groupId`)
- **Watch for the Kafka compaction trap** — compacted topics still have duplicates mid-cycle; your projector needs to handle last-writer-wins, not skip-if-seen
- **PostgreSQL** per the DB conventions RFC, partitioned if you expect >50M rows
- **Staleness indicator** on reports — the view is eventually consistent, so expose the lag

### What to watch out for

The main risk is **schema drift** — when source services evolve their events and your projector silently breaks. The Jackson enum trap you've already been bitten by (`READ_UNKNOWN_ENUM_VALUES_USING_DEFAULT_VALUE` without `@JsonEnumDefaultValue`) will bite again. Build explicit schema compatibility checks into the consumer.
