I've reviewed the SPINE and N26 architecture. Before I dive in, a few important observations:

**Your team (Investments/Lifecycle) doesn't own the auth system.** Auth is handled at the Ruthenium (API gateway) level with TOKEN/SCA modes, and the session infrastructure is likely owned by a platform/identity team. A migration this wide (6 downstream services, mobile, web, CI/CD) is a **company-wide initiative**, not a team-level story.

That said, here's a structured plan you could propose or contribute to:

---

## JWT Migration Plan

### Phase 0: Design & Alignment (2-3 weeks)
- **RFC** — Write a company-wide RFC (N26 loves these, see Database Conventions RFC pattern). Cover: token format, signing algorithm (RS256), key rotation strategy, refresh token lifecycle, revocation mechanism
- **Stakeholder alignment** — Platform team, mobile teams (iOS + Android), web, security, compliance (this is a bank — regulatory implications exist)
- **Token design** — Claims schema, expiry windows (short-lived access: 15min, refresh: 7-30 days), audience/issuer fields
- **Revocation strategy** — Pure JWT is stateless but you need revocation for account closures, AML seizures (see `account_lock_reason` in Carbonium). Options: short expiry + refresh, or a lightweight blacklist in Redis

### Phase 1: Infrastructure (2-3 weeks)
- **Auth service changes** — Issue JWT access + refresh token pairs alongside existing sessions
- **Key management** — RSA key pair generation, Vault integration (N26 uses Vault — see `static/databases/{engine}/{service}` pattern), rotation schedule
- **CI/CD for key rotation** — Automated rotation pipeline, overlap window where old+new keys are both valid
- **Ruthenium gateway** — Add JWT validation path in parallel to existing TOKEN auth. Ruthenium uses Envoy — this means adding a JWT auth filter in the Envoy config

### Phase 2: Dual-Mode (3-4 weeks)
- **Ruthenium accepts both** — Session tokens AND JWTs. Route-level config in the Kotlin DSL (`src/main/kotlin/de/tech26/ruthenium/config/gateway/services/`)
- **Downstream services** — The 6 services (Magneton, Magnemite, Phosphine, Acetate, Casein, Xanthine) should NOT need changes if Ruthenium handles auth and forwards a standard user context header. Verify each service extracts user identity from headers, not session state
- **Feature flag** — Gate JWT issuance behind a flag (N26 uses Statsig). Roll out to internal users first
- **Mobile apps** — Update auth flow to request + store JWT pair, handle refresh logic, fallback to session if JWT fails. This is the **longest pole** — app release cycles are slow

### Phase 3: Gradual Rollout (4-6 weeks)
- **Internal → % rollout** — Start with employees, then 1% → 10% → 50% → 100%
- **Monitoring** — Track auth failures in Datadog/OpenSearch. Watch for:
  - Token expiry storms (many tokens expiring simultaneously)
  - Refresh token race conditions on mobile (parallel requests)
  - Clock skew issues between services
- **Web app** — Update to use JWT with httpOnly secure cookies or Authorization header
- **Zero-downtime guarantee** — The dual-mode phase is your safety net. Never remove session support until 100% of traffic is JWT and stable for 2+ weeks

### Phase 4: Cleanup (2 weeks)
- Remove session issuance
- Remove dual-mode code in Ruthenium
- Archive session storage infrastructure
- Update Jarfish (test environment) to use JWT for simulated calls

### Key Risks

1. **Mobile release cycle** — iOS/Android app updates take 1-2 weeks to reach 95% adoption. Old app versions need session support indefinitely or a forced update
2. **Revocation latency** — Pure JWT means a compromised token is valid until expiry. For a bank, this may not pass compliance review without a blacklist
3. **Refresh token storage on mobile** — Secure enclave / keychain required. This is where security will push back hardest
4. **Cookie-monster tooling** — Your team's internal auth tooling (see `ops/cookie-auth-tooling.md`) will need updates to work with JWTs

### Honest Assessment

This is **not a 1-point story**. This is a quarter-long, multi-team initiative. If this came up in sprint planning as a 1-pointer, I'd flag it — either the scope needs to be narrowed (e.g., "write the RFC" or "add JWT validation to Ruthenium") or the estimate is off by an order of magnitude.
