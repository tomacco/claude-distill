Here's an implementation plan for session-to-JWT migration with refresh tokens:

---

## Phase 0: Foundation (pre-migration)

**Key generation & infrastructure:**
- Generate RSA-256 key pair for JWT signing
- Set up key rotation infrastructure (CI/CD secret rotation on schedule)
- Deploy JWKS endpoint in auth service (public keys for verification)
- Provision refresh token store (Redis or DB table — short-lived sessions → long-lived refresh tokens need revocation list)

**Auth service changes:**
- New endpoints: `POST /auth/token` (issue JWT + refresh), `POST /auth/refresh`, `POST /auth/revoke`
- JWT payload: `sub`, `exp` (short — 15min), `iat`, `jti`, `scope/roles`
- Refresh token: opaque, stored server-side, longer TTL (7-30d), bound to device fingerprint
- Token family tracking for refresh token rotation (detect reuse → revoke family)

---

## Phase 1: Dual-mode gateway (zero-downtime bridge)

**API Gateway:**
- Accept BOTH session cookie AND `Authorization: Bearer` header
- Priority: JWT > session (if both present, use JWT)
- Gateway validates JWT signature locally using JWKS (no auth service call on hot path)
- On session-only requests: pass through to existing session validation (no change for old clients)
- Propagate standardized `X-User-Id` / `X-User-Scopes` headers downstream

**Downstream services (all 6):**
- Stop reading session directly — read only gateway-injected headers
- This is the decoupling step: services become auth-mechanism-agnostic
- Deploy and validate independently per service

---

## Phase 2: Client migration

**Mobile (iOS + Android):**
- Store refresh token in Keychain/Keystore (NOT SharedPreferences/UserDefaults)
- Implement token refresh interceptor (401 → refresh → retry original request)
- Handle refresh token rotation (new refresh token on each use)
- Handle concurrent requests during refresh (queue or lock pattern)
- Graceful fallback: if refresh fails → re-authenticate

**Web:**
- Refresh token in `httpOnly`, `Secure`, `SameSite=Strict` cookie (NOT localStorage)
- Access token in memory only (JS variable, not persisted)
- Silent refresh via `/auth/refresh` before expiry (proactive, not reactive)
- Tab synchronization via BroadcastChannel for token state

---

## Phase 3: Hardening & cutover

**Security:**
- Refresh token revocation on: password change, logout, suspicious activity
- Rate limit on refresh endpoint
- JTI blocklist for compromised access tokens (Redis, short TTL = access token TTL)
- Audit logging on all token lifecycle events

**Key rotation CI/CD:**
- Rotate signing keys on schedule (e.g., 90 days)
- JWKS endpoint serves current + previous key (overlap window)
- Gateway validates against all keys in JWKS
- Automate via GitHub Actions or your CD pipeline

**Cutover:**
- Feature flag: disable session validation in gateway
- Monitor error rates on the flag flip
- Decommission session store after bake period

---

## Phase ordering & dependencies

```
Phase 0 ──→ Phase 1 ──→ Phase 2 ──→ Phase 3
                │            │
                │            ├── iOS (parallel)
                │            ├── Android (parallel)  
                │            └── Web (parallel)
                │
                ├── Service A (parallel)
                ├── Service B (parallel)
                └── ...Service F (parallel)
```

**Critical path:** Auth service (Phase 0) → Gateway dual-mode (Phase 1) → at least one client migrated (Phase 2) → cutover (Phase 3).

**Parallelizable:** All 6 downstream services (Phase 1), all 3 clients (Phase 2).

---

## Risks to front-load

1. **Clock skew** — JWT `exp` validation breaks if service clocks drift. Use NTP, add 30s leeway in validation.
2. **Token size** — JWTs in headers can exceed proxy limits if claims bloat. Keep payload minimal.
3. **Refresh token reuse detection** — if you DON'T implement token family tracking, a stolen refresh token is usable until expiry. This is non-negotiable for mobile.
4. **Logout semantics change** — sessions = server-side kill. JWT = client still has valid token until expiry. Decide: accept 15min window, or maintain a blocklist?
