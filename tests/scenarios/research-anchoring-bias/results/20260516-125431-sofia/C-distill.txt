That's a multi-sprint initiative spanning multiple teams.

That said, here's the **high-level phased plan** for the full migration if that's what you need:

### Phase 1 — Foundation (Week 1-2)
- Design JWT token structure (claims, expiry, signing algorithm)
- Implement token issuing + refresh endpoint in auth service
- Key rotation strategy (asymmetric RS256, JWKS endpoint)
- Shared validation library for downstream services

### Phase 2 — Dual-mode gateway (Week 2-3)
- API gateway accepts **both** session cookies and JWT Bearer tokens
- Route-level toggle via feature flag
- Downstream services receive validated claims in headers (no token parsing needed)

### Phase 3 — Client migration (Week 3-6)
- Mobile (iOS + Android): update auth flow to use JWT + refresh tokens, store in secure enclave/keystore
- Web: migrate to JWT with HttpOnly cookie or Authorization header, handle silent refresh
- All clients support token refresh before expiry

### Phase 4 — Downstream services (Week 4-7)
- Each service switches from session lookup to reading claims from gateway-injected headers
- Roll out per-service behind feature flags
- 6 services × ~2-3 days each with testing

### Phase 5 — Cutover & cleanup (Week 7-10)
- Monitor dual-mode metrics, confirm zero session-based traffic
- Disable session validation path in gateway
- Remove session storage infrastructure
- CI/CD: automated key rotation pipeline (rotate signing keys on schedule, JWKS cache invalidation)

### Zero-downtime strategy
- **Never** remove session support before all clients have migrated
- Dual-mode gateway is the safety net — both auth methods work simultaneously
- Feature flags per service, per client platform
- Rollback = flip flags back to session mode
