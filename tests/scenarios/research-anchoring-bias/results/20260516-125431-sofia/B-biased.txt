## Reality Check

This is **not** a 1-point story. A session-based → JWT migration touching an auth service, API gateway, 6 downstream services, 2 mobile apps, a web app, CI/CD, and requiring zero-downtime is a **multi-quarter initiative** for a team. Planning it in 2 hours is reasonable — executing it is months.

That said, here's the implementation plan:

---

## Phase 0: Foundation (Week 1-2)

**Goal:** JWT infrastructure without touching production traffic.

1. **Key management setup**
   - Generate RSA-256 key pair (or Ed25519 if infra supports it)
   - Store private key in vault (AWS Secrets Manager / HashiCorp Vault)
   - Publish JWKS endpoint (`/.well-known/jwks.json`) from auth service
   - Set up key rotation CI job (rotate every 90 days, overlap window of 7 days with `kid` header)

2. **Token design**
   - Access token: short-lived (15 min), contains `sub`, `roles`, `iat`, `exp`, `kid`
   - Refresh token: long-lived (7-30 days), opaque, stored server-side (DB or Redis)
   - Define revocation strategy: token blacklist (Redis TTL = access token lifetime) for immediate logout

3. **Auth service changes**
   - New endpoints: `POST /auth/token` (issue), `POST /auth/refresh`, `POST /auth/revoke`
   - Dual-mode: still creates sessions AND issues JWTs simultaneously

---

## Phase 1: Gateway + Downstream (Week 3-5)

**Goal:** Services can validate JWTs but still accept sessions.

4. **API Gateway (dual-mode validation)**
   - Accept both `Cookie: session=...` AND `Authorization: Bearer ...`
   - JWT validation: verify signature against JWKS, check `exp`, check blacklist
   - Forward decoded claims as headers (`X-User-Id`, `X-Roles`) to downstream
   - Session requests continue working unchanged

5. **Downstream services (6 services)**
   - Remove direct session validation logic
   - Trust gateway-injected headers (they already should if gateway is the trust boundary)
   - If any service calls auth service to validate sessions directly → replace with header trust
   - **Order:** Start with lowest-traffic service, end with critical path

---

## Phase 2: Clients (Week 5-8)

**Goal:** Apps use JWTs, with fallback.

6. **Mobile (iOS + Android)**
   - Store refresh token in Keychain/Keystore (encrypted)
   - Access token in memory only (never persisted)
   - Implement silent refresh: on 401, use refresh token, retry original request (queue concurrent requests)
   - Feature flag: `use_jwt_auth` — toggle per user cohort

7. **Web app**
   - Refresh token: `httpOnly`, `secure`, `sameSite=strict` cookie (NOT localStorage)
   - Access token: in-memory variable
   - Implement refresh interceptor (same pattern: 401 → refresh → retry)
   - Feature flag: same as mobile

---

## Phase 3: Migration + Cutover (Week 8-10)

**Goal:** Zero-downtime transition from sessions to JWT-only.

8. **Gradual rollout**
   - 1% → 10% → 50% → 100% of new logins get JWT (existing sessions still valid)
   - Monitor: latency, error rates, auth failures per service
   - Existing sessions naturally expire (don't force-invalidate)

9. **Kill switch**
   - If JWT error rate > threshold → gateway falls back to session validation
   - Feature flag at gateway level, not per-service

10. **Session deprecation**
    - After 100% on JWT for 2 full session-lifetime cycles (e.g., 2 weeks if sessions last 7 days)
    - Remove session creation from auth service
    - Remove session validation from gateway
    - Drop session store (Redis/DB table)

---

## Phase 4: Hardening (Week 10-12)

11. **Key rotation CI/CD**
    - Cron job generates new key pair, publishes to JWKS with new `kid`
    - Old key remains in JWKS for `max_access_token_lifetime` after rotation
    - Alert if JWKS has only 1 key (rotation failed)

12. **Monitoring & alerting**
    - Dashboard: refresh token usage rate, blacklist size, token validation latency
    - Alert: spike in refresh failures (indicates stolen/expired refresh tokens)
    - Alert: JWKS endpoint latency > 100ms (services cache, but cold start matters)

13. **Security hardening**
    - Refresh token rotation (new refresh token issued with each use, old one invalidated)
    - Token binding (optional): tie refresh token to device fingerprint
    - Rate limit on `/auth/refresh` per user

---

## Key Risks & Mitigations

| Risk | Mitigation |
|------|-----------|
| Clock skew causes premature expiry | Add 30s leeway in validation; use NTP |
| Refresh token theft | Rotation + device binding + short refresh lifetime for sensitive users |
| JWKS endpoint down | Services cache JWKS with 1hr TTL; gateway has in-memory cache |
| Key rotation breaks in-flight tokens | Overlap window; `kid` header; validate against ALL keys in JWKS |
| Mobile app update lag | Old app versions use sessions until force-update; gateway dual-mode stays until mobile adoption > 95% |
