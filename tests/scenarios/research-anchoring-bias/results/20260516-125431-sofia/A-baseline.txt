## JWT Migration Plan: Session-Based → JWT + Refresh Tokens

### Phase 0 — Design & Alignment (Week 1-2)

| Task | Owner |
|------|-------|
| RFC document: token format, claims schema, rotation strategy, revocation approach | Backend lead |
| Decide: opaque vs transparent refresh tokens (recommend opaque — stored server-side for revocation) | Architecture |
| Key management design: asymmetric (RS256/ES256) with JWKS endpoint | Security |
| Define token lifetimes: access (5-15min), refresh (7-30 days) | Security + Product |
| Migration strategy decision: dual-mode (both auth methods) during transition | All |
| Threat model: token theft, replay, refresh token rotation | Security |

**Key decision**: Access tokens are short-lived JWTs verified by downstream services without calling auth. Refresh tokens are opaque, stored in a DB, revocable.

---

### Phase 1 — Auth Service (Weeks 3-5)

1. **New endpoints:**
   - `POST /auth/token` — issue JWT access + opaque refresh token
   - `POST /auth/token/refresh` — rotate refresh token, issue new access token
   - `POST /auth/token/revoke` — revoke refresh token (logout)
   - `GET /.well-known/jwks.json` — public key endpoint

2. **Key rotation infrastructure:**
   - Generate RSA/EC key pairs, store in Vault/KMS
   - JWKS endpoint serves current + previous public key (overlap window)
   - CI/CD cron rotates signing key every 30 days, keeps previous key valid for access token max lifetime

3. **Refresh token storage:**
   - New table: `refresh_tokens(id, user_id, token_hash, device_id, expires_at, revoked_at)`
   - Implement refresh token rotation (one-use, issue new on refresh)
   - Detect reuse → revoke entire family (stolen token detection)

4. **Backward compat:**
   - Existing session endpoints remain functional (dual mode)
   - New login flow returns JWT + refresh token alongside session cookie

---

### Phase 2 — API Gateway (Weeks 4-6, overlaps Phase 1)

1. **Dual-mode validation:**
   - Accept both `Authorization: Bearer <jwt>` AND existing session cookie
   - JWT path: validate signature locally using cached JWKS (no auth service call per request)
   - Session path: existing behavior (call auth service)

2. **JWKS caching:**
   - Cache public keys with TTL (5 min), refresh on unknown `kid` in token header
   - Circuit breaker on JWKS fetch failure (use cached keys)

3. **Header propagation:**
   - Extract claims from JWT, forward as trusted headers to downstream services
   - Same headers currently forwarded from session validation → zero downstream change initially

---

### Phase 3 — Downstream Services (Weeks 6-8)

Since the gateway already forwards user context as headers, downstream services **don't need immediate changes**. However:

1. **Optional direct JWT validation** (for service-to-service calls that bypass gateway):
   - Shared library with JWKS client + token validation
   - Each service adds dependency, configures JWKS URL

2. **Service-to-service tokens:**
   - Machine-to-machine JWT (client credentials flow) for inter-service calls
   - Separate signing key, shorter lifetime (1-5 min)

3. **All 6 services:**
   - Add JWT validation library dependency
   - Integration tests with both auth modes
   - No business logic changes required if gateway handles propagation

---

### Phase 4 — Mobile & Web (Weeks 7-10)

**Mobile (iOS + Android):**
1. Token storage: Keychain (iOS) / EncryptedSharedPreferences (Android)
2. Auth interceptor: attach `Authorization: Bearer` header
3. 401 handler: attempt refresh, queue concurrent requests during refresh
4. Refresh failure → logout flow
5. Background refresh: proactively refresh when token is >80% expired

**Web:**
1. Access token: in-memory only (never localStorage — XSS risk)
2. Refresh token: httpOnly secure cookie (or BFF pattern)
3. Silent refresh via iframe/background fetch before expiry
4. Tab synchronization for logout

**All clients:**
- Dual mode: if JWT flow fails, fall back to session during migration
- Feature flag to control rollout percentage

---

### Phase 5 — Migration & Cutover (Weeks 10-13)

1. **Gradual rollout:**
   - 5% → 25% → 50% → 100% of new logins get JWT flow
   - Existing sessions remain valid until natural expiry
   - Monitor: auth service load, error rates, latency

2. **Metrics to watch:**
   - Auth service request volume (should drop as JWT validation is local)
   - 401 rates (shouldn't spike)
   - Refresh token usage patterns
   - Gateway latency (should improve — no auth call per request)

3. **Session deprecation:**
   - Stop issuing new sessions
   - Set hard expiry on remaining sessions (e.g., 30 days)
   - Remove session validation code from gateway

---

### Phase 6 — Hardening & Cleanup (Weeks 13-15)

- Remove dual-mode code paths
- Remove session tables/infrastructure
- Pen test the new auth flow
- Document runbook for key rotation failures
- Alerting on refresh token reuse (compromise indicator)

---

### CI/CD for Key Rotation

```
Scheduled pipeline (monthly):
1. Generate new key pair
2. Upload private key to Vault/KMS
3. Update JWKS endpoint (add new key, keep old)
4. Wait > max access token lifetime (15 min)
5. Mark old key as deprecated (still valid for verification)
6. After 24h grace period, remove old key from JWKS
```

---

### Zero-Downtime Guarantees

| Risk | Mitigation |
|------|-----------|
| Gateway can't reach JWKS | Cached keys + circuit breaker |
| Key rotation during traffic | Overlap window (old + new key both valid) |
| Client has expired access token | Refresh transparently, queue requests |
| Refresh token DB migration | Blue-green deploy, new table doesn't touch existing |
| Client not yet updated | Dual-mode gateway accepts sessions indefinitely until cutover |
| Refresh token stolen | Rotation detection → revoke family |
