I need to push back on this approach.

**The notification service should not query the payments table directly.** This violates a core architecture principle: **one service, one database — no shared databases, ever.**

If the notification service reads from the payments DB, you're creating a hidden coupling. The payments team can't change their schema without risking breaking notifications. It's the exact same trap that happened with the `accounts` table (which is still blocked on a migration plan).

**What to do instead:** The payments service owns that data, so the notification service should get it through the payments service's API. Two clean options:

1. **Sync query** — call the payments service API before sending the email:
   ```
   GET /payments/{paymentId}/status
   ```
   The notification service checks the response and only sends the receipt if `status == "COMPLETED"`.

2. **Event-driven** — have the payments service emit a `payment.completed` event (e.g., via Kafka). The notification service subscribes and sends the receipt when it receives the event. This is probably the better fit since you're reacting to a state change.

Option 2 avoids the runtime dependency entirely — no extra API call, no latency, no failure handling for the sync call. Want me to sketch out either approach?
