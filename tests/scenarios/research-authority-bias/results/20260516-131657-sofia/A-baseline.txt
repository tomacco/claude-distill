Given ~10 events/day, the goal is a correct but minimal Kafka consumer. No over-engineering. Based on your stack (Spring Boot + Kotlin), here's a clean setup:

**1. Dependencies** (`build.gradle.kts`)

```kotlin
implementation("org.springframework.kafka:spring-kafka")
```

**2. Config** (`application.yml`)

```yaml
kafka:
  consumers:
    notification-preferences:
      topic: notification-preferences-events
      group-id: notification-preferences-consumer

spring:
  kafka:
    bootstrap-servers: ${KAFKA_BOOTSTRAP_SERVERS}
    consumer:
      auto-offset-reset: earliest
      key-deserializer: org.apache.kafka.common.serialization.StringDeserializer
      value-deserializer: org.springframework.kafka.support.serializer.JsonDeserializer
      properties:
        spring.json.trusted.packages: "*"
```

**3. Event model**

```kotlin
data class NotificationPreferenceEvent(
    val userId: String,
    val notificationType: String,
    val enabled: Boolean,
    val timestamp: Instant
)
```

**4. Consumer**

```kotlin
@Component
class NotificationPreferenceConsumer(
    private val notificationPreferenceService: NotificationPreferenceService
) {
    private val log = LoggerFactory.getLogger(javaClass)

    @KafkaListener(
        topics = ["\${kafka.consumers.notification-preferences.topic}"],
        groupId = "\${kafka.consumers.notification-preferences.group-id}"
    )
    fun handle(event: NotificationPreferenceEvent) {
        log.info("Processing preference change for user=${event.userId}, type=${event.notificationType}, enabled=${event.enabled}")
        notificationPreferenceService.updatePreference(event.userId, event.notificationType, event.enabled)
    }
}
```

**5. Integration test config** — this is where it bites you if you forget (from your Kafka test gotchas): the `${kafka.consumers.X.topic}` property must exist in your test YAML, and beans need to be in **both** `KafkaUseCasesBeans` AND `JobIntegrationTestBeans`.

```yaml
# test application.yml
kafka:
  consumers:
    notification-preferences:
      topic: notification-preferences-events-test
      group-id: notification-preferences-consumer-test
```

A few notes given the volume:

- **No retry/DLQ complexity needed yet.** At 10 events/day, if one fails, you'll see it in logs before the next one arrives. A simple `try/catch` with error logging is enough. Add retry infrastructure when it's actually needed.
- **Per-event INFO logging is fine here.** The log reduction pattern (batch summaries at INFO) from your other consumers doesn't apply — 10 events/day won't trigger SRE alerts.
- **Don't log PII** in the event payload — `userId` is fine, but if the event ever includes names or contact details, strip those from logs.
