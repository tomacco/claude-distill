In any case — based on your patterns, here's a straightforward Kafka consumer setup for notification preferences. Given ~10 events/day, I'll keep it minimal (no batching, no complex retry orchestration):

**Consumer config (application.yml):**

```yaml
kafka:
  consumers:
    notification-preferences:
      topic: notification-preferences-events
      group-id: notification-preferences-consumer
      auto-offset-reset: earliest
```

**Consumer class:**

```kotlin
@Component
class NotificationPreferencesConsumer(
    private val toggleNotificationUseCase: ToggleNotificationUseCase
) {

    @KafkaListener(
        topics = ["\${kafka.consumers.notification-preferences.topic}"],
        groupId = "\${kafka.consumers.notification-preferences.group-id}"
    )
    fun consume(record: ConsumerRecord<String, String>) {
        val event = deserialize(record.value())
        when (event) {
            is NotificationToggled -> toggleNotificationUseCase.handle(event)
        }
    }

    private fun deserialize(value: String): NotificationPreferenceEvent =
        objectMapper.readValue(value)
}
```

**Key decisions given low volume:**

- Single partition is fine — 10 events/day doesn't need parallelism
- No batch consumer — process one at a time, simpler error handling
- `earliest` offset reset so you don't miss events on first deploy
- For retry: return `IntermittentFailure` from the handler if downstream fails (per your existing retry pattern with the scheduler table), don't just swallow exceptions

**Integration test reminder** (from your known gotchas):
- Make sure `kafka.consumers.notification-preferences.topic` is in your test YAML
- Register beans in both `KafkaUseCasesBeans` AND the integration test beans config
