In the meantime, here's a minimal Kafka consumer setup appropriate for your situation — keeping it simple given ~10 events/day, while following your team's established patterns:

**Key decisions based on what I know about your Kafka patterns:**
- Single partition is fine at this volume
- No need for batch-level log summarization (that's for high-volume consumers) — per-event INFO logging is appropriate here
- Return `IntermittentFailure` on transient errors to trigger retry (not `Success`)
- Don't log PII (user IDs are fine, but no names/emails in log lines)

Here's the consumer skeleton (assuming Spring Boot + Kotlin, your usual stack):

```kotlin
@Component
class NotificationPreferenceConsumer(
    private val updatePreferenceUseCase: UpdateNotificationPreferenceUseCase
) {
    private val log = LoggerFactory.getLogger(javaClass)

    @KafkaListener(
        topics = ["\${kafka.consumers.notification-preferences.topic}"],
        groupId = "\${kafka.consumers.notification-preferences.group-id}"
    )
    fun consume(record: ConsumerRecord<String, String>) {
        val event = parseEvent(record.value())
        log.info("Processing notification preference toggle for userId={}, channel={}", event.userId, event.channel)

        try {
            updatePreferenceUseCase.execute(event.userId, event.channel, event.enabled)
        } catch (e: Exception) {
            log.error("Failed to process notification preference event for userId={}", event.userId, e)
            throw e // let the error handler trigger retry
        }
    }

    private fun parseEvent(value: String): NotificationPreferenceEvent =
        objectMapper.readValue(value)
}
```
