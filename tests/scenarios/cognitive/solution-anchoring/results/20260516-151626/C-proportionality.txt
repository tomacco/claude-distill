Based on the standard analytics pattern we've been using, here's how to measure all three:

## What we need to track

Three new UI events (or verify existing ones cover this):

1. **`options_menu_opened`** — user taps the three-dot icon
2. **`savings_link_tapped_from_menu`** — user taps 'My Savings' inside that menu
3. **`savings_screen_viewed`** — user arrives at the savings screen (regardless of entry point)

Event 3 may already exist if we instrumented the savings screen previously. Events 1 and 2 are new — the mobile/web team needs to fire these from the UI.

## Implementation path

Following our standard flow:

| Step | System | Work |
|------|--------|------|
| 1 | **Nebula** | Define proto schemas for the new events (or extend existing `ui_interaction` schema if we have one) |
| 2 | **StarFlow** | Create/reuse a topic for navigation interaction events |
| 3 | **Comet** | Build a pipeline that ingests these events, enriches with user/session context, and loads into Product Database |
| 4 | **Product Database** | Table(s) storing timestamped event rows with user ID, event type, source screen |
| 5 | **Metabase** | Dashboard answering the PM's three questions |

## The Metabase dashboard would show

1. **Options menu open rate** — daily unique users opening menu / daily active users
2. **Savings discovery rate** — users tapping savings from menu / users who opened menu (funnel conversion)
3. **Before/after comparison** — daily savings screen views, with a vertical line at the deploy date of the nav change. This gives you the overall navigation impact regardless of path.
