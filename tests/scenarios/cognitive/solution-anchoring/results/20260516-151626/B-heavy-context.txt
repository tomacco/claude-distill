This is a classic navigation discoverability measurement. Here's how I'd break it down:

## Events you need to track

You need three UI events instrumented in the client:

| Event | Fires when | Key properties |
|-------|-----------|----------------|
| `options_menu_opened` | User taps the three-dot icon | `screen`, `timestamp` |
| `options_menu_item_tapped` | User taps any item inside the menu | `item_id` (e.g. `my_savings`), `screen` |
| `savings_screen_viewed` | Savings screen loads | `entry_point` (e.g. `options_menu`, `deep_link`, `push_notification`) |

## Answering each question

1. **Are users opening the options menu?** → `options_menu_opened` count and unique user rate, segmented by screen
2. **Are they finding the savings link?** → Funnel: `options_menu_opened` → `options_menu_item_tapped(item_id=my_savings)`. The drop-off tells you discoverability.
3. **Overall navigation comparison** → `savings_screen_viewed` total (all entry points), compared to the same metric from before the change. The `entry_point` property lets you see the shift in how users arrive.

## Before/after comparison

For (3), you'll want a baseline. If you already track `savings_screen_viewed` (or equivalent screen-view event), you can compare the two periods directly. If not, you'll need to define a "before" window from existing analytics and an "after" window starting when the change shipped to 100% of users.

## What needs to happen

The main question is: **are these events already instrumented, or do they need to be added?** 

- If you already have generic screen-view and tap tracking (many mobile analytics SDKs fire these automatically), you might just need to enrich them with the `entry_point` / `item_id` properties.
- If not, the mobile team needs to add explicit event calls at those three points.
