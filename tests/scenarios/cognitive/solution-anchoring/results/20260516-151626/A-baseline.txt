Here's how I'd structure the measurement for this nav change:

## Events to instrument

**1. Options menu opens**
```
event: options_menu_opened
properties: { screen: "main_nav", trigger: "three_dot_tap" }
```

**2. Savings link tapped from options menu**
```
event: savings_link_tapped
properties: { source: "options_menu" }
```

**3. Savings screen viewed** (you likely already have this)
```
event: screen_viewed
properties: { screen: "savings_dashboard" }
```

## Metrics that answer each question

| Question | Metric | How to compute |
|----------|--------|----------------|
| Are users opening the options menu? | Options menu open rate | `options_menu_opened` / DAU (or per session) |
| Are they finding savings from there? | Savings tap-through rate | `savings_link_tapped (source=options_menu)` / `options_menu_opened` |
| Overall navigation comparison | Savings screen arrival rate | `screen_viewed (screen=savings_dashboard)` / DAU — compare pre vs post sprint |

## What to watch for

- **Drop in savings screen views** — the most important signal. If overall arrivals drop significantly, the link is effectively hidden.
- **Funnel drop-off**: menu opened → savings tapped. If the menu gets opened but savings isn't tapped, the link might be buried among too many items.
- **Compensating paths**: users might find savings through other routes (search, deep links, shortcuts). Tag `savings_link_tapped` with `source` on ALL entry points so you can see if traffic shifted rather than disappeared.

## Before/after comparison

You need a clean baseline. Pull `screen_viewed (screen=savings_dashboard)` for the 2-4 weeks before the change, same metric after. Segment by:
- New vs returning users (new users won't have muscle memory for the old location)
- Platform (iOS/Android if applicable)

If you already have the `screen_viewed` event, question (3) is answerable today from existing data — you just need a date split around the release. Questions (1) and (2) require the new events to be instrumented.
