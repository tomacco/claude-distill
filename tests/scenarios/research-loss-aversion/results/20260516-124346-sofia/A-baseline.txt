You have every reason to remove it. "Better safe than sorry" is how dead code lives forever.

Eight months deprecated with a replacement in production is more than generous. Here's how to move forward confidently instead of indefinitely:

**Verify before removing (not instead of removing):**
1. Check access logs — is anything still hitting `/v1/legacy-notifications`? If traffic is zero or near-zero, you're done deliberating.
2. If there IS residual traffic, identify the caller. It's likely a forgotten integration or a client that never migrated. Reach out, give them a hard deadline, then remove.
3. Check if there's a deprecation header or response warning already being sent — if so, callers have been warned.

**The cost of keeping it is real:**
- Every engineer who touches that area has to understand code that shouldn't exist
- Tests for dead code slow down CI
- It creates ambiguity about which endpoint is canonical
- It accumulates security surface for no value

**If you want a safety net:** return `410 Gone` for a sprint before full deletion. Anyone still calling it gets an unmistakable signal, and you get one last chance to catch stragglers in monitoring.
